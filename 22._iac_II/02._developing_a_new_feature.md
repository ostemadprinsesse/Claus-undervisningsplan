# Developing a New Feature

**Type**: Individual/Group

**Motivation**: This document describes the recommended workflow for implementing a new feature in your group project. It covers branch protection, automated code review with CodeRabbit, and the pull request process.

The example feature used here is adding a teardown script for your infrastructure — but the approach applies to any feature.

---

## Prerequisites

**Kanban**: When developing a new feature you should add this to your task list (Kanban), and assign the feature to one of your groups members.

**Branch protection**:  Before starting, make sure your repository has branch protection rules configured. If you have not done that yet, follow the guide here:

- [Branch protection rules](../06._CI_github_actions/exercises/branch_protection_rules.md)

The minimum recommended rules for your main branch:
- Require a pull request before merging
- Require at least 1 approval (2 is better)
- Require status checks to pass before merging

---

## Step 1 — Set Up CodeRabbit

[CodeRabbit](https://coderabbit.ai) is an AI-powered code review tool that automatically reviews your pull requests and leaves comments on the diff. It integrates directly with GitHub.

### 1.1 Install the GitHub App

1. Go to [coderabbit.ai](https://coderabbit.ai) and sign in with your GitHub account.
2. Click **Add to GitHub** and select your group's repository (or organization).
3. Grant the requested permissions and confirm the installation.

CodeRabbit will now automatically post a review summary on every pull request you open.

### 1.2 Add a configuration file (optional but recommended)

You can customize CodeRabbit's behavior by adding a `.coderabbit.yaml` file to the root of your repository:

```yaml
# .coderabbit.yaml
reviews:
  auto_review:
    enabled: true
    drafts: false         # do not review draft PRs
  poem: false             # skip the poem in the summary
```

Commit and push this file to your main branch. CodeRabbit will pick it up automatically.

---

## Step 2 — Create a Feature Branch

Never commit directly to `master`/`main`. Create a dedicated branch for the feature.

```bash
# Make sure you are up to date with main
git checkout master
git pull origin master

# Create and switch to a new feature branch
git checkout -b feature/teardown-script
```

Name your branch descriptively: `feature/<short-description>`. Keep it lowercase with hyphens.

---

## Step 3 — Implement the Feature

Work on the feature in your new branch. Commit your changes in small, logical steps with clear commit messages.

```bash
# Example: after writing infrastructure/teardown.sh
git add infrastructure/teardown.sh
git commit -m "Add teardown script for Azure VM infrastructure"
```

Good commit messages:
- Are written in the imperative: *"Add"*, *"Fix"*, *"Update"* — not *"Added"* or *"Adding"*
- Describe *what* the change does and *why* if it is not obvious

Push your branch to GitHub when you are ready for review:

```bash
git push origin feature/teardown-script
```

---

## Step 4 — Open a Pull Request

1. Go to your repository on GitHub.
2. GitHub will usually show a banner: **"Compare & pull request"** — click it. Otherwise go to **Pull requests → New pull request**.
3. Set the **base branch** to `master` and the **compare branch** to `feature/teardown-script`.
4. Fill in the PR description:
   - What does this feature do?
   - How was it tested?
   - Any notes for the reviewer?
5. Click **Create pull request**.

---

## Step 5 — CodeRabbit Reviews the PR

Within a minute or two, CodeRabbit will post an automated review on your pull request. It will:

- Summarize the changes
- Point out potential issues, missing error handling, or style concerns
- Leave inline comments on specific lines

Read the feedback. When CodeRabbit raises a valid concern, you have three options for addressing it:

**Option 1: Apply the "Committable suggestion" directly on GitHub**

CodeRabbit leaves a "Committable suggestion" block collapsed inside the comment. Click it and commit the fix with a single button — no need to touch your local code at all. This is the fastest path for small issues.

**Option 2: Ask CodeRabbit to autofix**

Type in the PR comment box to apply all suggestions at once:

```
@coderabbitai autofix
```

Or for stacked pull requests:

```
@coderabbitai autofix stacked pr
```

The bot will attempt to push the fix as a commit directly to your branch. Note: aliases `@coderabbitai auto-fix` or `@coderabbitai auto fix` are also supported.

**Option 3: Send it back to the author of the PR**

This is the most pedagogically valuable option in a team setting. Leave a comment requesting they fix it before merging:

```
@clbokea please address the CodeRabbit suggestions before this is merged
```

You can also configure a branch protection rule so PRs cannot be merged until CodeRabbit's review passes — that forces everyone to engage with the feedback rather than ignore it.

You do not need to act on every comment. Use your judgement. If you disagree, reply to the comment to explain your reasoning.

---

## Step 6 — Human Review and Approval

At least one (ideally two) group members should review the pull request:

1. Go to the **Files changed** tab and read through the diff.
2. Leave comments on anything unclear or that should be changed.
3. When satisfied, click **Review changes → Approve**.

The author should not approve their own PR.

If changes are requested, the author commits the fixes to the same branch and asks for a re-review.

---

## Step 7 — Merge the Pull Request

Once all required approvals are in and all status checks pass:

1. Click **Merge pull request**.
2. Use **Squash and merge** if you want a clean single commit on main, or **Merge commit** to preserve the full history — agree on this as a group and be consistent.
3. After merging, delete the feature branch (GitHub offers a button for this).

```bash
# Clean up locally as well
git checkout master
git pull origin master
git branch -d feature/teardown-script
```

---

## Summary

| Step | Action |
|------|--------|
| 1 | Branch protection rules are in place |
| 2 | CodeRabbit is installed on the repository |
| 3 | Create `feature/<name>` branch from master |
| 4 | Implement and commit the feature |
| 5 | Push and open a pull request |
| 6 | CodeRabbit posts automated review |
| 7 | Group members approve after review |
| 8 | Merge and delete the feature branch |

This workflow ensures that no code reaches `master` without being reviewed — both by an AI tool and by at least one human. It also creates a clear history of what was changed and why.
